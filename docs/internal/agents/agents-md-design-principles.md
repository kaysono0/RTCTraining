# AGENTS.md Design Principles

本文档总结 `AGENTS.md` 的设计准则，供其他项目复用。它不是某个项目的完整模板，而是一套判断标准。

## 1. 核心定位

`AGENTS.md` 是给 AI Coding Agent 的项目地图。

它应该回答：

- 这个项目是什么。
- 哪些方向不能做。
- 代码和文档入口在哪里。
- 哪些工程规则必须遵守。
- 改完后如何验证。

它不应该承载完整开发手册、详细历史记录或长篇设计说明。

## 2. 地图优先

根目录 `AGENTS.md` 应控制长度，优先保留导航和硬规则。

适合放在 `AGENTS.md`：

- 项目定位。
- 禁止方向。
- 关键模块位置。
- 已确认的架构决策。
- 编码硬规则。
- 测试和启动命令。
- 提交前检查。
- 详细文档入口。

适合放到 `docs/`：

- 详细 API 设计。
- 阶段计划。
- UI 状态说明。
- 实验方案。
- 历史开发记录。
- 排障细节。

判断标准：

```text
AI 不知道就容易写错代码 -> 放 AGENTS.md
AI 不知道只是需要多查一点 -> 放 docs，并在 AGENTS.md 里放入口
```

## 3. 先写项目边界

项目定位应放在最前面。Agent 首先需要知道项目要成为什么，以及不要成为什么。

建议包含：

- 一句话项目定位。
- 主要开发闭环或核心工作流。
- 3 到 5 个优先级关键词。
- 明确不做的能力。

“不做什么”很重要。它能阻止 Agent 把学习项目写成生产系统，把本地工具写成云平台，把简单实验写成复杂架构。

## 4. 固定决策要显式化

已经确认的工程决策应直接列出，避免 Agent 反复询问或自行改方向。

常见类型：

- 服务监听地址和端口。
- 支持的平台、浏览器、运行环境。
- 数据隔离字段。
- API 响应格式。
- 前后端访问边界。
- 技术栈限制。
- 首版和增强版的能力边界。

这些规则要写成可执行约束，避免只写抽象原则。

## 5. 仓库结构写到可定位

仓库结构不需要完整树形图，重点是告诉 Agent 去哪里找什么。

推荐写法：

```text
src/server/：后端服务入口和 handler
src/domain/：纯业务逻辑，不依赖 Web 框架
templates/：服务端页面模板
static/：前端脚本和样式
tests/：单元测试和端到端测试
docs/：设计文档和阶段计划
```

目录说明应和真实职责绑定，不写泛泛的“存放源码”。

## 6. 规则要能改变行为

`AGENTS.md` 中的规则应服务于减少错误。

有效规则：

- Store 层保持纯 Python，不导入 Web 框架。
- 所有 API 响应必须使用统一 envelope。
- 页面状态必须暴露测试钩子。
- Dashboard 不直接访问跨端口服务。

低价值规则：

- 代码要清晰。
- 注意可维护性。
- 保持良好风格。
- 合理处理异常。

抽象要求应转成具体做法、文件边界或测试要求。

## 7. 命令要统一入口

Agent 需要稳定的构建、启动和验证命令。复杂环境操作应封装到脚本或 Makefile。

推荐包含：

```bash
make setup
make run
make test-unit
make test-e2e
make test
```

如果项目有多个服务，应写清：

- 每个服务的启动命令。
- 默认监听地址。
- 健康检查或 smoke check。
- 常用 curl 示例。

长排障说明放到独立文档，例如 `docs/agents/verification.md`。

## 8. 验证规则要靠前

Agent 的工作目标不是只改代码，而是改完能验证。

建议写清：

- 新行为先写测试。
- 相关测试必须运行。
- UI 变更要跑浏览器或 E2E。
- 服务变更要启动服务并做 smoke check。
- 无法验证时要说明原因。

验证命令越稳定，Agent 自主修复能力越强。

## 9. 把长期记忆移出根文档

项目演进记录、最近测试结果、历史决策背景、提权经验等内容有价值，但不适合长期堆在根目录 `AGENTS.md`。

推荐拆分：

```text
AGENTS.md                                  # 项目地图和硬规则
docs/agents/project_memory.md              # 项目记忆和历史经验
docs/agents/verification.md                # 验证和排障说明
docs/superpowers/plans/<phase>.md          # 阶段计划
docs/<feature>.md                          # 详细设计
```

这样根文档保持短，细节仍可被 Agent 按需读取。

## 10. 维护方式

`AGENTS.md` 应由 bad case 驱动迭代。

当 Agent 犯错时，判断原因：

- 缺少全局规则：补到 `AGENTS.md`。
- 缺少模块细节：补到对应 `docs/`。
- 缺少验证方式：补到验证文档或 Makefile。
- 规则可自动检查：优先加测试或 lint。

不要把所有经验都写进根文档。根文档越长，关键规则越容易被忽略。

## 11. 推荐章节模板

可从以下结构开始：

```md
# <Project> Agent Guide

## 1. 项目定位
## 2. 阅读入口
## 3. 当前阶段
## 4. 仓库结构
## 5. 固定决策
## 6. 后端规则
## 7. 前端规则
## 8. 数据和接口规则
## 9. 开发节奏
## 10. 常用命令
## 11. 本地入口
## 12. 提交前检查
## 13. AGENTS.md 维护规则
```

按项目类型删减章节。小项目不需要完整结构，保留定位、目录、规则、命令即可。
