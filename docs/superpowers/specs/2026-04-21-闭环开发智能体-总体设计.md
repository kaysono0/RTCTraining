# RTCTraining 闭环开发智能体总体设计

## 1. 目标

为 RTCTraining 设计一个受约束、可审计的闭环开发智能体。它能够读取低风险任务契约，创建隔离工作区，生成开发计划，修改代码，运行必要测试，根据失败结果进行有限修复，并输出可供人工审查的产物。

这个智能体不是完全自治的仓库维护者，而是任务契约执行器。它的权限来自任务 JSON、仓库策略、命令白名单、路径白名单和人工审批。

## 2. 设计依据

本设计基于 `RTCTraining_整理版.md` 中的约束：

- RTCTraining 的核心价值是 WebRTC 实验闭环：连接建立、stats 采集、实验控制、CSV 固化、Dashboard 对比和自动化任务收口。
- `automation/` 是独立的低风险自动开发 runner，不属于 WebRTC 主运行路径。
- runner 必须先校验路径策略、命令策略和任务风险等级。
- runner 应输出 plan、patch、transcript、task report、morning summary 等审计产物。
- runner 不应修改 `data/` 下的实验产物。
- runner 应采用最小必要验证，不默认运行最重的测试。

## 3. 非目标

- 不自动合并主分支。
- 不任意执行 shell 命令。
- 不修改任务契约之外的路径。
- 不做公网生产级多用户平台。
- 不做 TURN / SFU / 认证 / 长期存储等 RTC 生产能力。
- 不允许无限自修复。
- 不允许隐藏执行过程，所有模型请求、响应、命令和结果都必须落盘。

## 4. 推荐路线

采用 worktree 闭环模式。

每个任务使用独立 git worktree。智能体读取任务契约后，先通过策略门禁，再生成计划和 patch。patch 通过 diff 校验后才允许应用。应用后运行 required checks。如果测试失败，智能体读取失败日志并判断是否允许在任务边界内修复。修复次数达到上限仍失败时，任务标记为 failed 或 blocked，并保留所有产物。

相比 patch-only，这条路线具备真实开发和真实测试闭环。相比全自治 Agent，它的风险更可控。

## 5. 闭环流程

```text
任务契约
  -> 策略门禁
  -> Worktree 管理
  -> 上下文加载
  -> 计划生成
  -> 代码修改
  -> Diff 校验
  -> 测试选择
  -> 测试执行
  -> 失败分析
  -> 有限修复
  -> 最终自审
  -> 产物导出
  -> 人工审批
```

## 7. 任务契约

任务放在：

```text
automation/tasks/ready/<task_id>.json
```

示例：

```json
{
  "id": "fix-dashboard-empty-stats",
  "goal": "修复 Dashboard 在 stats 数据为空时的展示问题。",
  "context_files": [
    "src/dashboard/server.py",
    "src/dashboard/templates/index.html",
    "tests/test_dashboard.py"
  ],
  "allowed_paths": [
    "src/dashboard/**",
    "tests/test_dashboard*.py",
    "docs/**"
  ],
  "forbidden_paths": [
    "data/**",
    "cert.pem",
    "key.pem",
    ".env",
    ".venv/**"
  ],
  "acceptance": [
    "无 stats 数据时 Dashboard 返回 HTTP 200。",
    "页面展示空状态而不是抛异常。",
    "Dashboard 相关测试通过。"
  ],
  "required_checks": [
    "pytest tests/test_dashboard.py"
  ],
  "optional_checks": [
    "pytest tests/test_dashboard_realtime.py"
  ],
  "risk_level": "low",
  "mode": "worktree",
  "max_repair_attempts": 3
}
```

## 8. 策略模型

全局策略位于：

```text
automation/config/policy.json
```

策略包括：

- 命令白名单。
- 全局禁止路径。
- 风险等级行为。
- 上下文大小限制。
- patch 大小限制。
- 默认修复次数。

风险等级建议：

```text
low
  允许计划、patch、测试和有限修复。

medium
  允许计划和 patch 草案，不自动应用，不自动运行命令。

high
  只允许生成计划。
```

全局禁止路径建议：

```text
data/**
automation/artifacts/**
.git/**
.venv/**
cert.pem
key.pem
.env
```

`automation/artifacts/**` 只能由 `ArtifactStore` 写入，模型生成的 patch 不允许修改审计产物。

## 9. 核心模块

### 9.1 `task_loader.py`

读取任务契约，校验必填字段，拒绝路径穿越，并在 `ready`、`running`、`done`、`failed` 之间移动任务文件。

### 9.2 `policies.py`

加载全局策略，校验任务路径、命令、风险等级，并决定任务是否允许 patch、运行测试或只生成计划。

### 9.3 `worktree_manager.py`

为每个任务创建隔离 worktree。默认路径为：

```text
.automation/worktrees/<task_id>
```

默认分支名为：

```text
automation/<task_id>
```

如果当前目录不是 git 仓库，worktree 模式应阻塞并说明原因。

### 9.4 `context_loader.py`

只读取任务声明的 `context_files` 和允许的关联文档。它不读取 `data/`、证书、虚拟环境和 artifacts。

### 9.5 `model_gateway.py`

统一模型调用接口，首版必须支持 `stub` 后端。所有请求和响应都写入 transcript，便于审计和测试。

### 9.6 `planner.py`

生成结构化计划。计划必须包含 summary、files_to_change、tests_to_run、risk_notes。计划中出现越权文件或遗漏 required checks 时必须拒绝。

### 9.7 `developer.py`

根据计划和上下文生成 unified diff。它不直接写文件，不直接运行命令。

### 9.8 `diff_validator.py`

解析 diff 并校验：

- 所有文件都在 allowed_paths 内。
- 没有命中 forbidden_paths。
- 没有二进制 patch。
- 没有修改 `automation/artifacts/**`。
- patch 大小没有超过限制。

### 9.9 `test_selector.py`

根据变更路径选择推荐测试，同时保留任务契约中的 required checks。

### 9.10 `test_runner.py`

运行白名单内的测试命令，并保存 stdout、stderr、exit code、开始时间和耗时。

### 9.11 `failure_analyzer.py`

分析失败日志，输出 `repair`、`blocked` 或 `unrelated`。只有 `repair` 允许进入下一轮修复。

### 9.12 `reviewer.py`

最终自审，检查 acceptance、required checks、diff 路径、生成文件和报告证据。

### 9.13 `artifact_store.py`

统一写入 plan、patch、diff、命令日志、transcript、任务报告和晨报。

## 10. 状态机

```text
ready
  -> policy_rejected
  -> running

running
  -> planned
  -> patched
  -> testing
  -> repairing
  -> passed
  -> blocked
  -> failed

passed
  -> done

blocked
  -> failed

failed
  -> failed archive
```

## 11. 成功标准

任务只有满足以下条件才能标记为 `done`：

- 最终 patch 能干净应用。
- 所有修改文件都在允许路径内。
- 没有触碰禁止路径。
- 报告中逐条检查 acceptance。
- 所有 required checks 都通过。
- optional checks 通过，或有明确跳过原因。
- patch、测试日志、transcript、报告都已落盘。

如果测试缺失、依赖缺失、仓库不完整或无法创建 worktree，任务应标记为 `blocked`，不能标记为成功。

## 12. 测试策略

智能体自身应优先使用 stub 模型测试，不依赖真实模型。

单元测试覆盖：

- 任务契约校验。
- 路径策略匹配。
- 命令白名单。
- diff 校验。
- 测试选择规则。
- artifact 写入。
- 状态机转换。

集成测试覆盖：

- 低风险任务在 stub 模式下成功完成。
- 越权 patch 被拒绝。
- 测试失败触发一次修复。
- 缺失测试命令导致 blocked。
- medium 任务停在 patch 前。
- high 任务只生成计划。

## 13. MVP 范围

第一版应实现：

1. JSON 任务读取。
2. 静态策略加载。
3. `patch-only` 和 `worktree` 模式。
4. `stub` 模型后端。
5. plan JSON 校验。
6. unified diff 校验和应用。
7. required checks 执行。
8. 有限修复循环。
9. patch、测试日志、transcript、report 输出。

第一版不需要追求真实模型效果。先把闭环机制做稳，再接入真实模型。

## 14. 人工审查流程

每个任务完成后生成：

```text
automation/artifacts/reports/<task_id>.md
automation/artifacts/patches/<task_id>.patch
automation/artifacts/test-runs/<task_id>/*.json
automation/artifacts/transcripts/<task_id>/*.json
```

人工审查者阅读报告和 patch，决定是否应用、合并或退回。合并动作始终由人控制。
