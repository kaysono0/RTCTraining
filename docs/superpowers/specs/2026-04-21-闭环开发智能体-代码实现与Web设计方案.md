# RTCTraining 闭环开发智能体：代码实现与 Web 设计详细方案

## 1. 项目目标

本方案面向 `RTCTraining` 的 `automation/` 子系统，目标是设计一个“自己开发、自己测试、自己生成审计报告”的闭环开发智能体。

它不是一个完全自治的通用编程助手，而是一个受任务契约和策略引擎约束的本仓库专用开发执行器。它的核心能力是：

- 从任务契约读取目标、上下文、允许路径、验收标准和必跑测试。
- 在隔离工作区中生成计划、修改代码、运行测试。
- 根据失败日志进行有限次数自修复。
- 产出 patch、测试日志、模型 transcript、任务报告和晨报。
- 提供 Web 控制台，让人可以查看任务状态、审批风险、检查 diff 和测试证据。

本工程的第一优先级不是“模型写代码多聪明”，而是“每一次自动开发行为都可约束、可复现、可审计、可回滚”。

## 2. 系统边界

### 2.1 包含范围

- 任务契约 JSON 管理。
- 路径、命令、风险等级策略校验。
- `patch-only` 和 `worktree` 两种执行模式。
- 模型网关抽象，首版支持 `stub`，后续接入 OpenAI / Anthropic / 智谱等后端。
- 计划生成、diff 生成、diff 校验、patch 应用。
- 测试命令选择与执行。
- 失败分析与有限修复循环。
- 报告、patch、测试日志、transcript、晨报落盘。
- Web 控制台查看和操作任务。

### 2.2 不包含范围

- 不自动合并到主分支。
- 不自动推送远端仓库。
- 不执行任意 shell 命令。
- 不修改 `data/` 下的实验产物。
- 不绕过任务契约修改未授权路径。
- 不作为公网多用户平台。
- 不实现生产级权限系统，首版只服务本地或可信局域网。

## 3. 总体架构

```text
Web Console
  -> HTTP API
  -> Automation Server
  -> Orchestrator
  -> Policy Gate
  -> Worktree Manager
  -> Planner
  -> Developer
  -> Diff Validator
  -> Test Runner
  -> Failure Analyzer
  -> Reviewer
  -> Artifact Store

Task Files
  -> automation/tasks/ready
  -> automation/tasks/running
  -> automation/tasks/done
  -> automation/tasks/failed

Artifacts
  -> automation/artifacts/plans
  -> automation/artifacts/patches
  -> automation/artifacts/reports
  -> automation/artifacts/test-runs
  -> automation/artifacts/transcripts
```

核心思路：

- CLI 负责可靠执行。
- Web 负责观察、审批和人工介入。
- 文件系统是首版的事实存储。
- JSON 和 Markdown 作为可读、可迁移的主产物。
- 模型不直接拥有执行权，所有模型输出都经过结构校验和策略校验。

## 5. 数据模型设计

### 5.1 任务契约

文件路径：

```text
automation/tasks/ready/<task_id>.json
```

字段：

```json
{
  "id": "fix-dashboard-empty-stats",
  "title": "修复 Dashboard 空 stats 页面",
  "goal": "当 stats 数据为空时，Dashboard 仍能稳定返回页面并展示空状态。",
  "context_files": [
    "src/dashboard/server.py",
    "src/dashboard/templates/index.html",
    "tests/test_dashboard.py"
  ],
  "allowed_paths": [
    "src/dashboard/**",
    "tests/test_dashboard*.py",
    "docs/**"
  ],
  "forbidden_paths": [
    "data/**",
    "cert.pem",
    "key.pem",
    ".env",
    ".venv/**"
  ],
  "acceptance": [
    "无 stats 数据时 Dashboard 返回 HTTP 200。",
    "页面展示空状态而不是抛异常。",
    "Dashboard 相关测试通过。"
  ],
  "required_checks": [
    "pytest tests/test_dashboard.py"
  ],
  "optional_checks": [
    "pytest tests/test_dashboard_realtime.py"
  ],
  "risk_level": "low",
  "mode": "worktree",
  "max_repair_attempts": 3,
  "notes": "不要修改 WebRTC 主信令逻辑。"
}
```

### 5.2 任务运行状态

运行时状态由 `automation/artifacts/reports/<task_id>.json` 保存一份机器可读版本，同时生成 Markdown 报告。

```json
{
  "task_id": "fix-dashboard-empty-stats",
  "status": "passed",
  "phase": "done",
  "risk_level": "low",
  "mode": "worktree",
  "started_at": "2026-04-21T15:20:00+08:00",
  "finished_at": "2026-04-21T15:24:10+08:00",
  "changed_files": [
    "src/dashboard/server.py",
    "tests/test_dashboard.py"
  ],
  "checks": [
    {
      "command": "pytest tests/test_dashboard.py",
      "status": "passed",
      "exit_code": 0,
      "duration_ms": 1840,
      "log_path": "automation/artifacts/test-runs/fix-dashboard-empty-stats/001.json"
    }
  ],
  "artifacts": {
    "plan": "automation/artifacts/plans/fix-dashboard-empty-stats.json",
    "patch": "automation/artifacts/patches/fix-dashboard-empty-stats.patch",
    "report": "automation/artifacts/reports/fix-dashboard-empty-stats.md"
  }
}
```

### 5.3 策略配置

文件路径：

```text
automation/config/policy.json
```

示例：

```json
{
  "allowed_command_prefixes": [
    ["pytest"],
    ["python3", "-m", "pytest"],
    ["python3", "-m", "compileall"],
    ["git", "diff"],
    ["git", "status"],
    ["git", "worktree"]
  ],
  "global_forbidden_paths": [
    "data/**",
    "automation/artifacts/**",
    ".git/**",
    ".venv/**",
    "cert.pem",
    "key.pem",
    ".env"
  ],
  "risk_levels": {
    "low": {
      "allow_patch": true,
      "allow_checks": true,
      "allow_repair": true
    },
    "medium": {
      "allow_patch": false,
      "allow_checks": false,
      "allow_repair": false
    },
    "high": {
      "allow_patch": false,
      "allow_checks": false,
      "allow_repair": false
    }
  },
  "max_context_bytes": 200000,
  "max_patch_bytes": 80000,
  "default_max_repair_attempts": 3,
  "command_timeout_seconds": 300
}
```

## 6. 后端代码实现方案

### 6.1 `task_loader.py`

职责：

- 扫描 `automation/tasks/ready/*.json`。
- 读取并解析 JSON。
- 校验必要字段。
- 拒绝路径穿越。
- 移动任务文件到 `running`、`done`、`failed`。

核心接口：

```python
class TaskLoader:
    def list_ready_tasks(self) -> list[TaskContract]:
        ...

    def load_task(self, task_id: str) -> TaskContract:
        ...

    def move_task(self, task_id: str, target_state: str) -> Path:
        ...
```

必要测试：

- 缺少 `goal` 的任务被拒绝。
- `context_files` 包含 `../secret` 被拒绝。
- 同名任务不能重复运行。
- 任务能从 `ready` 移到 `running`。

### 6.2 `policies.py`

职责：

- 加载全局策略。
- 判断任务是否允许 patch、运行测试、修复。
- 校验命令是否匹配白名单。
- 校验路径是否命中允许路径且不命中禁止路径。

核心接口：

```python
class PolicyEngine:
    def validate_task(self, task: TaskContract) -> PolicyDecision:
        ...

    def is_path_allowed(self, task: TaskContract, path: str) -> bool:
        ...

    def is_command_allowed(self, command: str) -> bool:
        ...
```

路径匹配建议使用 `fnmatch`，不要手写脆弱的字符串包含判断。

### 6.3 `worktree_manager.py`

职责：

- 为任务创建独立工作区。
- 创建 `automation/<task_id>` 分支。
- 在任务结束后保留或清理 worktree。

首版策略：

- 如果当前目录不是 git 仓库，自动降级为 `patch-only`，并在报告中标记 `blocked_for_worktree`。
- 如果 git 工作区不干净，不创建 worktree，除非用户明确选择只生成 patch。

核心接口：

```python
class WorktreeManager:
    def prepare(self, task: TaskContract) -> WorktreeInfo:
        ...

    def cleanup(self, task: TaskContract, keep: bool = True) -> None:
        ...
```

### 6.4 `context_loader.py`

职责：

- 读取 `context_files`。
- 读取必要文档。
- 读取允许路径下的文件列表。
- 收集轻量 git 信息。

限制：

- 不读取 `data/**`。
- 不读取 `.venv/**`。
- 不读取证书和密钥。
- 不读取超过 `max_context_bytes` 的内容。

输出结构：

```json
{
  "task": {},
  "files": [
    {
      "path": "src/dashboard/server.py",
      "content": "..."
    }
  ],
  "repo": {
    "branch": "main",
    "status": "clean"
  }
}
```

### 6.5 `model_gateway.py`

职责：

- 统一模型调用接口。
- 支持 `stub` 后端。
- 保存每一次请求与响应。
- 校验模型输出是否为期望 JSON 或 unified diff。

首版必须先实现 `stub`：

```python
class StubModelGateway:
    def complete_json(self, prompt_name: str, payload: dict) -> dict:
        ...

    def complete_text(self, prompt_name: str, payload: dict) -> str:
        ...
```

为什么先做 `stub`：

- 可以先验证闭环框架。
- 可以写稳定测试。
- 不依赖外部网络和模型费用。
- 后续接真实模型时不会影响核心流程。

### 6.6 `planner.py`

职责：

- 生成结构化开发计划。
- 验证计划没有越权文件。
- 验证计划包含所有必跑测试。

计划格式：

```json
{
  "summary": "修复 Dashboard 空数据分支，并补充回归测试。",
  "files_to_change": [
    "src/dashboard/server.py",
    "tests/test_dashboard.py"
  ],
  "tests_to_run": [
    "pytest tests/test_dashboard.py"
  ],
  "risk_notes": [
    "不修改 data 目录。",
    "不修改 WebRTC 信令逻辑。"
  ]
}
```

### 6.7 `developer.py`

职责：

- 根据计划和上下文请求模型生成 unified diff。
- 不直接修改文件。
- 不直接运行命令。

输出必须是标准 diff：

```diff
diff --git a/src/dashboard/server.py b/src/dashboard/server.py
--- a/src/dashboard/server.py
+++ b/src/dashboard/server.py
@@ -10,6 +10,8 @@
 ...
```

### 6.8 `diff_validator.py`

职责：

- 解析 diff 中的文件路径。
- 校验所有路径都在 `allowed_paths` 内。
- 校验没有命中 `forbidden_paths`。
- 拒绝二进制 patch。
- 拒绝修改 `automation/artifacts/**`。
- 拒绝超大 patch。

如果 diff 不合法，任务直接进入 `blocked`，不进入测试阶段。

### 6.9 `test_selector.py`

职责：

- 根据变更文件推导推荐测试。
- 保留任务契约中的 `required_checks`。
- 可将推荐测试追加为 optional checks。

规则：

```text
src/webrtc/mesh_handlers.py 或 src/webrtc/chat_server.py
  -> pytest tests/test_mesh_signaling.py
  -> pytest tests/test_mesh_room_lifecycle.py

src/webrtc/stats_store.py 或 src/webrtc/stats_handlers.py
  -> pytest tests/test_unit.py
  -> pytest tests/test_rtcp_analysis.py

src/dashboard/**
  -> pytest tests/test_dashboard.py
  -> pytest tests/test_dashboard_realtime.py

src/webrtc/static/** 或 src/webrtc/chat_real.html
  -> pytest tests/test_mesh_playwright.py
  -> pytest tests/test_webrtc_test_session_playwright.py
```

如果测试文件不存在，不能伪造通过，应在报告中标记为 `missing_check`。

### 6.10 `test_runner.py`

职责：

- 运行通过白名单校验的测试命令。
- 捕获 stdout、stderr、exit code、耗时。
- 将结果保存为 JSON。

结果格式：

```json
{
  "command": "pytest tests/test_dashboard.py",
  "exit_code": 0,
  "status": "passed",
  "started_at": "2026-04-21T15:20:00+08:00",
  "duration_ms": 1840,
  "stdout": "...",
  "stderr": ""
}
```

### 6.11 `failure_analyzer.py`

职责：

- 阅读失败日志。
- 判断失败是否与本次 patch 相关。
- 给出 `repair`、`blocked`、`unrelated` 三种结论。

结构：

```json
{
  "outcome": "repair",
  "summary": "测试失败来自空 stats 分支未处理 None。",
  "likely_files": [
    "src/dashboard/server.py"
  ],
  "repair_guidance": "在渲染前将 None stats 归一为空列表。"
}
```

只有 `outcome = repair` 才允许进入下一轮修复。

### 6.12 `reviewer.py`

职责：

- 最终自审。
- 对照 acceptance。
- 对照 required checks。
- 对照 diff 路径。
- 生成 Markdown 报告。

报告结构：

```markdown
# 任务报告：fix-dashboard-empty-stats

## 结果

passed

## 变更文件

- src/dashboard/server.py
- tests/test_dashboard.py

## 验收项

- [x] 无 stats 数据时 Dashboard 返回 HTTP 200。
- [x] 页面展示空状态而不是抛异常。
- [x] Dashboard 相关测试通过。

## 测试

- pytest tests/test_dashboard.py: passed

## 产物

- Patch: automation/artifacts/patches/fix-dashboard-empty-stats.patch
- 测试日志: automation/artifacts/test-runs/fix-dashboard-empty-stats/
- Transcript: automation/artifacts/transcripts/fix-dashboard-empty-stats/
```

### 6.13 `orchestrator.py`

职责：

- 串联完整状态机。
- 提供 CLI。
- 提供给 Web API 调用的服务接口。

CLI 设计：

```bash
python3 -m automation.runner.orchestrator list
python3 -m automation.runner.orchestrator run fix-dashboard-empty-stats
python3 -m automation.runner.orchestrator run-next
python3 -m automation.runner.orchestrator summarize
```

状态机：

```text
ready
  -> policy_check
  -> running
  -> planning
  -> patching
  -> validating_diff
  -> applying_patch
  -> testing
  -> repairing
  -> reviewing
  -> done / failed / blocked
```

## 7. Web 控制台设计

## 7.1 Web 产品定位

Web 控制台不是给最终 RTC 实验用户使用的页面，而是给开发者早晨检查夜间自动化任务、审批中风险任务、查看测试证据和下载 patch 使用的工程控制台。

它应该强调：

- 当前有没有任务在跑。
- 哪些任务成功、失败、阻塞。
- 每个任务到底改了什么。
- 测试是否真的运行过。
- 哪些地方需要人工介入。

## 7.2 信息架构

```text
Dashboard
  -> 任务队列概览
  -> 最近运行结果
  -> 风险和阻塞提醒

Tasks
  -> Ready
  -> Running
  -> Done
  -> Failed

Task Detail
  -> 契约
  -> 计划
  -> Diff
  -> 测试
  -> Transcript
  -> 报告

Artifacts
  -> Patches
  -> Reports
  -> Test Runs
  -> Transcripts

Policy
  -> 命令白名单
  -> 禁止路径
  -> 风险等级

Morning Summary
  -> 今日摘要
  -> 需要人工处理的问题
```

## 7.3 首页 Dashboard

首页应是一个密集但清晰的工程状态面板。

主要区域：

1. 顶部状态条
   - 当前 runner 状态：idle / running / blocked。
   - 当前任务 ID。
   - 最近一次运行时间。
   - 策略版本。

2. 指标区
   - Ready 任务数。
   - Running 任务数。
   - Passed 任务数。
   - Failed / Blocked 任务数。
   - 最近 24 小时测试通过率。

3. 任务列表
   - 任务 ID。
   - 标题。
   - 风险等级。
   - 模式：patch-only / worktree。
   - 当前阶段。
   - 最后更新时间。
   - 操作按钮：查看、运行、停止、归档。

4. 阻塞区
   - 测试缺失。
   - 命令不在白名单。
   - 路径越权。
   - patch 应用失败。
   - 模型输出格式错误。

视觉风格：

- 使用浅色工程控制台风格。
- 背景为接近白色的灰色。
- 状态色克制使用：绿色通过、红色失败、黄色阻塞、蓝色运行。
- 卡片只用于任务项和状态摘要，不做大面积装饰。
- 表格优先，方便扫描。

## 7.4 任务详情页

任务详情页是 Web 控制台最重要的页面。

布局：

```text
左侧：任务导航
右侧：详情内容

Tabs:
  契约 | 计划 | Diff | 测试 | Transcript | 报告
```

### 契约 Tab

展示任务 JSON 的结构化视图：

- goal
- context_files
- allowed_paths
- forbidden_paths
- acceptance
- required_checks
- optional_checks
- risk_level
- mode

需要高亮：

- 命中全局禁止路径的项。
- 不存在的 context file。
- 不在白名单的 required check。

### 计划 Tab

展示模型生成的计划：

- Summary。
- Files to change。
- Tests to run。
- Risk notes。

如果计划被策略拒绝，要展示拒绝原因。

### Diff Tab

展示 unified diff。

首版可以使用纯文本 diff，后续再做 side-by-side diff。

功能：

- 按文件折叠。
- 新增行绿色背景。
- 删除行红色背景。
- 文件名旁边显示是否允许修改。
- 提供下载 patch 按钮。

### 测试 Tab

展示每条测试命令：

- 命令。
- 状态。
- exit code。
- 耗时。
- stdout。
- stderr。

交互：

- stdout / stderr 默认折叠。
- 失败日志自动展开。
- 可复制命令。

### Transcript Tab

展示模型调用记录：

- prompt 名称。
- 请求时间。
- 响应时间。
- 输入摘要。
- 输出摘要。
- 原始 JSON 下载。

注意：首版本地可信环境可以展示完整 transcript。未来如果包含敏感信息，需要增加脱敏层。

### 报告 Tab

渲染 Markdown 任务报告。

报告必须能回答：

- Agent 做了什么。
- 改了哪些文件。
- 跑了哪些测试。
- 哪些测试通过或失败。
- 哪些验收项满足。
- 有没有人工需要继续处理的问题。

## 7.5 新建任务页

首版可以提供一个表单，用来生成任务 JSON。

字段：

- Task ID。
- 标题。
- 目标。
- 上下文文件。
- 允许路径。
- 禁止路径。
- 验收标准。
- 必跑测试。
- 可选测试。
- 风险等级。
- 执行模式。
- 最大修复次数。

交互要求：

- 每个多值字段用可增删的列表输入。
- 风险等级使用单选或 segmented control。
- 执行模式使用 `patch-only` / `worktree` 切换。
- 保存前在右侧实时预览 JSON。
- 保存后写入 `automation/tasks/ready/<task_id>.json`。

## 7.6 策略页

策略页用于查看 `automation/config/policy.json`。

首版建议只读，不提供在线编辑，避免误改安全边界。

展示：

- 命令白名单。
- 全局禁止路径。
- 风险等级行为。
- patch 大小限制。
- 上下文大小限制。
- 命令超时。

如果后续支持编辑，需要强制生成策略变更报告。

## 7.7 晨报页

晨报页展示夜间任务总结。

内容：

- 总任务数。
- 成功数。
- 失败数。
- 阻塞数。
- 每个任务的结果。
- 需要人工介入的问题。
- patch 下载入口。

示例：

```markdown
# RTCTraining 自动开发晨报

## 总览

- Ready: 0
- Done: 3
- Failed: 1
- Blocked: 2

## 需要人工处理

- fix-dashboard-empty-stats: 通过，等待 review。
- improve-mesh-timeout-test: blocked，测试文件不存在。
- refactor-stats-store: failed，第三轮修复后仍有断言失败。
```

## 13. 关键风险与处理

### 13.1 模型越权修改

处理：

- diff 应用前强制校验路径。
- 任何命中禁止路径的 patch 直接拒绝。
- 模型不能直接写文件。

### 13.2 测试被绕过

处理：

- `required_checks` 必须运行并通过。
- 测试文件缺失时标记 blocked。
- 不能用模型解释替代测试结果。

### 13.3 命令执行风险

处理：

- 命令必须命中白名单。
- 不支持 shell 拼接命令。
- 命令执行设置超时。
- stdout / stderr 全量落盘。

### 13.4 审计产物被篡改

处理：

- `automation/artifacts/**` 禁止出现在模型 patch 中。
- 只能由 `ArtifactStore` 写入。
- 每个产物记录生成时间和任务 ID。

### 13.5 当前目录不是 git 仓库

处理：

- `worktree` 模式标记 blocked。
- `patch-only` 模式仍可生成 plan 和 patch。
- 报告中必须明确说明无法创建 worktree。

## 14. 验收标准

MVP 完成时应满足：

- 可以创建任务 JSON。
- 可以通过 CLI 列出 ready 任务。
- 可以校验路径和命令策略。
- 可以用 stub 模型生成 plan。
- 可以校验并应用合法 diff。
- 可以拒绝越权 diff。
- 可以运行 required checks。
- 可以根据失败进入一次或多次修复。
- 可以生成 patch、测试日志、transcript、Markdown 报告。
- 可以通过 Web 查看任务、diff、测试日志和报告。
- required checks 未通过时任务不能标记 done。

## 15. 最终形态

这个工程最终应该形成一个本地开发自动化控制台：

```text
人写任务契约
  -> Agent 按策略开发
  -> Agent 跑测试
  -> Agent 自修复
  -> Agent 输出证据
  -> 人审 patch
  -> 人决定是否合并
```

它与 RTCTraining 主系统的关系是：

- WebRTC 主系统负责“RTC 实验闭环”。
- Dashboard 负责“实验观测闭环”。
- Automation Agent 负责“工程开发闭环”。

三者合在一起，构成一个既能做 RTC 实验，又能沉淀工程改进任务，并用自动化 runner 低风险推进的训练型仓库。
